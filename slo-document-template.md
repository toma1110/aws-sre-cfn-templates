---

## SLI 定義（Service Level Indicators）

> **SLI = サービス品質の "ものさし"**。何を計測するか。

| # | SLI | 説明 | 計測方法 |
|---|---|---|---|
| 1 | 可用性 | エラーなく返せたリクエストの割合 | `1 - (5xxエラー数 / 総リクエスト数)` |
| 2 | レイテンシ p99 | 100リクエスト中99番目に遅い応答時間 | ALB の `TargetResponseTime` の p99 統計 |

---

## SLO 目標値（Service Level Objectives）

> **SLO = どこまで品質を保証するかの "約束"**。

| # | SLI | 目標値 | 計測窓 |
|---|---|---|---|
| 1 | 可用性 | **99.5%** 以上 | 過去 28 日 |
| 2 | レイテンシ p99 | **500ms** 以下 | 過去 28 日 |

---

## エラーバジェット

> **エラーバジェット = SLO を達成しつつも許容できる失敗の量**。

| 項目 | 値 |
|---|---|
| SLO（可用性） | 99.5% |
| エラーバジェット | 0.5% / 28日 |
| 28日間のリクエスト数想定 | 約 X,000,000 |
| 許容失敗リクエスト数 | 約 XX,000 リクエスト |
| ダウンタイム換算 | 約 **3.36 時間** / 28日 |

---

## エラーバジェットポリシー

> 残量が変化したときに **誰がどう動くか** を事前に決めておく。
> インシデント時に「どう判断するか」で迷わないために重要。

| 残量 | 判断基準 | 行動 |
|---|---|---|
| **100% 〜 50%** | 健全 | 通常通り新機能リリースを進める |
| **50% 以下** | 警戒 | 新機能リリースを慎重に検討、信頼性改善タスクの優先度を上げる |
| **20% 以下** | アラート発報 | 原因調査を最優先、レビュー体制を強化、リリース速度を意識的に下げる |
| **0%（バジェット枯渇）** | 緊急 | 機能開発を一時停止し、信頼性改善に集中。ポストモーテムを実施 |

## 計測方法（CloudWatch Metric Math）

### 可用性 SLI

```
Metric Math:
  m1 = ALB の HTTPCode_Target_5XX_Count（Sum, 1時間）
  m2 = ALB の RequestCount（Sum, 1時間）
  e1 = (1 - m1 / m2) * 100
```

### レイテンシ p99

```
Metric: AWS/ApplicationELB / TargetResponseTime
Statistic: p99
Period: 1 時間
```

### エラーバジェット残量

```
Metric Math:
  budget_consumed = SUM(m1 over 28d) / SUM(m2 over 28d) * 100
  budget_remaining = (0.5 - budget_consumed) / 0.5 * 100   # %
```
