# ポストモーテム テンプレート

> **Blameless（責任追及しない）** ポストモーテムの考え方で記載してください。
> 個人の責任ではなく「仕組み」と「再発防止」に焦点を当てます。

---

## ① 要約（Summary）

> 忙しいステークホルダーが最初に読む部分。3〜5文でコンパクトに。

- **何が起きたか**: <例: ALB の 5xx エラー率が 12% に達し、約 18 分間サービス全体が不安定になりました>
- **影響**: <例: 全ユーザーの約 30% がチェックアウト処理に失敗。決済成立件数が前週比 -42%>
- **対応時間**: <例: 検知から復旧まで 25 分（14:03 〜 14:28 JST）>

---

## ② 影響範囲（Impact）

| 項目 | 内容 |
|---|---|
| 発生時刻（JST） | YYYY-MM-DD HH:MM 〜 HH:MM |
| 影響を受けたサービス | 例: チェックアウトAPI、決済サービス |
| 影響を受けたユーザー | 例: 約 12,000 人 / 全体の 30% |
| 影響を受けた地域 | 例: 全リージョン / ap-northeast-1 のみ |
| 業績影響 | 例: 売上機会損失 約 ¥XXX,000 / SLA違反: あり |

## ③ タイムライン（Timeline）

> **タイムゾーンは冒頭で宣言**（このタイムラインは **JST** 基準）。
> CloudWatch ログ（UTC）と Slack（JST）が混在しないよう注意。

| 時刻（JST） | 担当 | アクション |
|---|---|---|
| 14:03 | CloudWatch | アラート受信（CPU 95% 超過） |
| 14:05 | @担当者A | インシデント宣言 / 専用チャンネル `#inc-2026-XX-XX` 作成 / 調査開始 |
| 14:08 | @担当者A | CloudWatch にて ALB 5xx エラー急増を確認 |
| 14:11 | @担当者A | Session Manager で EC2 接続 / `stress-ng` プロセスを特定 |
| 14:13 | @担当者B | Run Command にて `killall stress-ng` 実行 |
| 14:15 | @担当者A | CPU が正常値に回復 / 5xx エラー収束を確認 |
| 14:17 | @担当者A | ✅ インシデント解決宣言 / Slack に投稿 |

## ④ 根本原因（Root Cause）

### Contributing Factors（寄与要因）

> 障害は単一の原因ではなく複数の要因が重なって発生する。まず列挙する。

1. <例: 負荷試験用の `stress-ng` プロセスが本番環境で停止されないまま残っていた>
2. <例: CPU 使用率の閾値アラームは設定済みだったが、対応 SOP が共有されていなかった>
3. <例: アラート通知が Slack の汎用チャンネルに集約され、見逃された>

### 5 Whys（最重要要因への深掘り）

> 最も重要な要因について「なぜ」を5回繰り返す。

- **問題**: <例: stress-ng プロセスが本番環境で動き続けていた>
- なぜ①: <例: 負荷試験後にプロセスが終了されなかった>
- なぜ②: <例: 負荷試験ジョブに自動 cleanup 処理が無かった>
- なぜ③: <例: 負荷試験スクリプトが「手動で叩いて Ctrl+C で止める」前提だった>
- なぜ④: <例: スクリプトが個人の作業メモから流用されたもので、運用標準化されていなかった>
- なぜ⑤: <例: 負荷試験の標準手順書（Runbook）が存在しなかった>
